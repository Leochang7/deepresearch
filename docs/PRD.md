# DeepResearch Agent PRD

## 1. 项目背景

复杂研究任务通常不是一次问答可以完成的。它需要问题拆解、资料检索、证据整理、跨步骤记忆、事实核查、报告合成和质量评估。普通 RAG 或单 Agent 系统容易出现以下问题：

- 长链路执行中上下文丢失。
- 子任务之间缺少依赖管理和并发控制。
- 报告事实错误、引用缺失或逻辑跳跃。
- 结果好坏依赖主观判断，缺少可复现评测。

DeepResearch Agent 的目标是实现一个面向开放式深度研究任务的多智能体协作系统，用工程化方式解决长链路研究任务的执行、修复和评测问题。

## 2. 产品定位

DeepResearch Agent 是一个研究报告生成系统，而不是通用聊天机器人。用户输入一个复杂研究问题，系统输出：

- 结构化研究报告。
- 支撑报告的引用和证据片段。
- 子任务执行过程与状态 trace。
- Red-Blue 审查与修复记录。
- 自动评测结果。

## 3. 目标用户

- 需要完成技术调研、产业分析、竞品研究、论文综述的个人开发者或研究者。
- 希望学习多 Agent 编排、RAG、评测体系和 LLM 工程实践的开发者。
- 面试或作品集场景下，需要展示复杂 Agent 系统工程能力的求职者。

## 4. 核心目标

### 4.1 产品目标

- 支持复杂研究问题从输入到报告输出的端到端流程。
- 支持任务拆解、并发执行、失败重试和动态 replan。
- 支持共享记忆，减少长链路任务的信息损失。
- 支持 Red-Blue 对抗审查与自动修复。
- 支持基础评测指标和可复现实验脚本。

### 4.2 非目标

MVP 阶段不追求：

- 完整 Web 产品形态。
- 大规模分布式 Agent 调度。
- 商业级爬虫系统。
- 覆盖所有数据源。
- 完全自动保证事实正确。

## 5. 核心用户流程

1. 用户输入研究问题、期望报告深度和约束条件。
2. Planner 将问题拆解为多个子任务，并形成 DAG。
3. Executor 根据依赖关系并发执行子任务。
4. Research Agent 通过统一 Retriever 接口检索资料、抽取证据、生成子任务摘要。
5. Memory 保存证据、摘要、来源和冲突信息。
6. Synthesizer 汇总子任务结果，生成完整报告。
7. Red Agent 从事实性、逻辑一致性和引用质量审查报告。
8. Blue Agent 对报告执行新增、删除、修改、验证等修复。
9. Judge 或 Evaluator 对最终报告打分，并输出执行日志。

## 6. 功能需求

### 6.1 任务规划

- 输入复杂研究问题，输出结构化任务计划。
- 每个任务包含：任务 ID、目标、依赖、输入、输出格式、优先级。
- 支持将任务计划表达为 DAG。
- 支持在任务失败或信息不足时触发 replan。

### 6.2 DAG 执行引擎

- 基于 `asyncio` 实现异步任务执行。
- 基于 `asyncio.Semaphore` 控制最大并发数。
- 支持任务状态机：
  - `PENDING`
  - `READY`
  - `RUNNING`
  - `SUCCEEDED`
  - `FAILED`
  - `SKIPPED`
  - `RETRYING`
  - `REPLANNING`
  - `CANCELLED`
- 支持单任务超时、批量失败 replan、全局超时强制合成。

### 6.3 Research Agent

- 根据子任务目标生成 3-5 个检索 query。
- 通过统一 Retriever 接口检索资料，MVP 支持本地资料集、Tavily 真实 Web 搜索和 MiMo 原生搜索。
- 提取关键事实、证据片段、来源 URL 和发布时间。
- 输出结构化 JSON，便于后续合成和评测。
- 支持不同 LLM 后端的热切换。

### 6.3.1 Retriever

- 提供统一资料获取接口，避免 Research Agent 绑定具体搜索服务、MCP 工具或模型原生搜索能力。
- MVP 支持：
  - `LocalDatasetRetriever`：从本地 Markdown/JSONL 资料集读取和召回资料。
  - `WebSearchRetriever`：默认通过 Tavily 获取候选网页、标题、摘要和 URL。
  - `MiMoSearchRetriever`：调用 MiMo v2.5 Pro 原生搜索并转换为统一资料格式。
- MVP 使用 `httpx + trafilatura` 对网页正文做轻量抓取和清洗。
- 后续支持：
  - `BrowserRetriever`：抓取网页正文并做正文清洗。
  - `MCPRetriever`：通过 MCP 接入外部工具和知识源。
  - `ModelNativeSearchRetriever`：接入模型厂商原生搜索能力，作为 fallback 或对比实验。
- Retriever 输出必须包含标题、URL、来源类型、正文、检索时间和元数据。

### 6.4 共享记忆

- 保存任务级摘要、证据、引用、冲突和最终结论。
- 基于 Milvus 支持向量检索、标量过滤和相似证据召回，帮助后续 Agent 找回相关信息。
- 写入前做去重。
- 标记潜在矛盾信息，交由后续审查或消解。

### 6.5 报告合成

- 生成结构化 Markdown 报告。
- 报告必须包含摘要、背景、分主题分析、结论、局限性和引用。
- 对关键结论附带 `[E12]` 形式的 evidence id 引用。
- 保留证据来源和子任务 trace。

### 6.6 Red-Blue 对抗修复

- Red Agent 审查报告问题：
  - 事实性错误。
  - 逻辑不一致。
  - 引用缺失或引用不支持结论。
  - 过度推断。
  - 结构混乱。
- Blue Agent 执行修复动作：
  - `ADD`
  - `DELETE`
  - `MODIFY`
  - `VERIFY`
- 支持多轮修复，但需要设置最大轮数。
- 支持评分收敛和震荡检测，避免无限循环。

### 6.7 评测体系

- 支持规则指标：
  - 引用覆盖率。
  - 空引用率。
  - 报告结构完整度。
  - 子任务完成率。
  - 失败重试次数。
- 支持 LLM-as-Judge 评分：
  - 事实准确性。
  - 逻辑一致性。
  - 覆盖度。
  - 引用质量。
  - 可读性。
- 支持实验结果导出为 JSON 和 Markdown。

## 7. 非功能需求

- 可复现：同一输入可以保存配置、执行 trace 和输出。
- 可观察：每个任务状态变化必须可记录。
- 可替换：LLM、Retriever、Memory、Evaluator 应通过接口解耦。
- 可测试：核心状态机、DAG 调度、JSON fallback 和评测指标必须有单元测试。
- 可降级：部分子任务失败时，系统仍能生成带局限性说明的报告。

## 8. 验收标准

MVP 通过以下标准即视为完成：

- 可以通过 CLI 输入一个研究问题并生成 Markdown 报告。
- Planner 能生成至少 5 个子任务组成的 DAG。
- Executor 能按依赖关系并发执行任务。
- Retriever 至少支持本地资料集、Tavily 真实 Web 搜索、MiMo 原生搜索、网页正文抓取和 mock 搜索后端。
- Memory 能将证据片段、摘要向量和元数据写入 Milvus，并按 `run_id`、`task_id`、`source_type` 等字段过滤检索。
- Red-Blue 至少完成一轮审查和修复。
- Evaluator 能输出基础规则指标。
- 示例任务、输出报告、执行 trace 可在仓库中复现。
